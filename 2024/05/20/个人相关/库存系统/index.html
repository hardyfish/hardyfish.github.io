库存域

名词解释

名词 含义
恶拍 恶意拍下商品，但是不付款的行为，此行为会导致商品少卖
超卖 被付款了的商品比库存商品要多，造成交易订单无法正常履约
少卖 实际库存比透出的库存要多，造成实际有货但是显示无货的现象。

库存组成

库存服务为 SKU 、活动 Offer、直充 等场景提供库存能力。商品绑定库存 id 后，交易在特定节点通过库存 id 申请扣减或回补。

库存分为3个层级

  • 库存记录：记录库存id，可售 O、预扣 W、占用 S 的当前数量。
  • 库存流水：通过外部 id + 库存 id，唯一确认的一系列操作的当前状态。
  • 库存流水记录：每次库存操作的详情，用于审计。

库存形式：除普通库存外，未来可能还有日历库存、渠道库存、地理库存等。

交易库存扣减策略

库存服务为 SKU 、活动 Offer、直充 等场景提供库存能力。商品绑定库存 id 后，交易在特定节点通过库存 id 申请扣减或回补。

库存实体关系：

实体 作用 分片键 幂等键 数据字段
库存 Inventory 库存数量记录 库存 id 库存 id 可售 O、预扣 W、占用 S 的数量
库存事务 Inventory Transaction 库存一系列相关操作的集合 库存 id 外部单据 id + 库存 id 可售 O、预扣 W、占用 S 的数量
库存记录 Inventory Log 库存相关操作集合中的一次操作 库存 id 外部单据 id + 操作类型 + 库存事务 id 可售 O、预扣 W、占用 S 的数量

库存可以理解为所有库存记录的累积视图，库存事务可以理解为某个操作集合的库存记录的累积视图

示例

如某物料现有库存 17 个，通过单据 A 申请预扣库存 10 个，随后通过子单据 B1 确认扣减库存 5 个、B2 再次确认扣减库存 2 个，并释放剩余库存 3 个；

其中的幂等键：

  • 库存事务：单据 A id + 库存 id
  • 库存记录 1：单据 A id + 预扣 + 库存事务 id
  • 库存记录 2：单据 B1 id + 扣减 + 库存事务 id
  • 库存记录 3：单据 B2 id + 扣减 + 库存事务 id
  • 库存记录 4：单据 A id + 释放 + 库存事务 id

拍减

下单即扣减库存，容易造成少卖

付减

付款才扣减库存，容易造成超卖

预扣

预扣结合拍减和付减，相当于两者的结合：

  1. 设置一个小于订单超时时间的预扣超时时间，将订单对库存(预扣)占用时间限制在合理范围内
  2. 下单时，只进行预扣
  3. 如果迟迟不进行付款，会提前释放预扣库存
  4. 付款时，真正进行扣减

预扣避免不了超卖的情况，可以根据需要退款或者发货。

库存优化方向

优化的核心在于控制好对同一个库存操作的TPS

目前一次下单(拍减模式)过程中对库存可能的操作来源有

  1. 预扣库存
  2. 预扣失败后同步释放库存
  3. 付款成功后扣减库存（在仅涉及到内部支付时扣减库存动作可以认为与预扣库存几乎是同步的）
  4. 用户主动取消订单

方案1

仅在 inventory 层做热点限流，仍保留预扣失败后的同步释放库存，好处是资源（库存）可以同步释放，但是整体的下单QPS会有些限制

方案2

在 inventory 层做热点限流，同时在预扣失败后异步释放库存（延时消息 3-5 min，并增加一定的随机），好处是可以减少释放库存的操作从而提高下单的QPS，同时同步释放可能会失败，延时消息可以保证尽早释放该库存，但是库存资源不会立即释放给购买用户

方案3

在 inventory 层做热点限流，预扣库存失败后同步释放库存的请求超过限流后，发送延时消息，通过延时消息异步释放库存，好处是可以尽可能的同步释放可能锁定的库存，需要底层限流能力的建设

库存多级缓存查询一致性

背景

库存服务作为底层核心服务，是典型的读写流量均较大的特殊场景，历史上峰值读流量达 1000w qps，峰值写流量达 1w qps。

因为针对读写流量均较大，且存在大量数据库行锁，前期引入了多级缓存体系、Inventory Hint 等手段，成功应对了春晚口播、辅导续报、斑马大促等场景。

随着库存服务使用的逐渐深入，对于后端赠课等短时间大量混合读写的场景，虽然库存服务可以保证下单的一致性，但是会造成“显示有库存但是赠课失败”的使用体验损失，故写本方案，提供解决办法。

方案

针对库存查询接口，通过提供层级选项，并针对选项提供不同的限流、熔断、SLA 策略，满足不同场景下，对库存查询实时性的诉求。

针对后台查询等场景，可以指定更高级别，以获更高的实时性。

新增参数 realTimeLevel:

序号 值 含义 限流情况 数据来源
0 DEFAULT 默认，同 LOW 同 LOW 同 LOW
1 LOW 低 无 LocalCache
2 MEDIUM 中 单业务线 100 QPS Redis
3 HIGH 高，暂未开放 单业务线 10 QPS DB
