一.系统概述

斑马营销系统是斑马电商最核心系统之一，承接了整个斑马上游业务方所有的营销诉求

1.功能概述

1.1 主要功能

1.2 使用业务方

口语课，AI课，积分商城，钻石商城，活动礼盒等（受限于个人业务知识）

2.非功能概述

  1. 匹配营销规则接口：99%响应时间80ms，qps0～20之间
  2. 查询积分商城营销活动接口：99%响应时间20ms，qps0～20之间
  3. 查询营销活动接口：99%响应时间20ms，qps0～10之间

二.系统架构与整体设计

1.整体架构

1.1 说明

后台运营

  1. 运营通过营销管理后台设置营销活动，选择参与营销的商品（营销范围），设置营销活动，营销规则，营销方式等信息，设置完成后发布上线
  2. 运营听过营销管理后台设置营销优惠券活动，选择给一批用户，发送优惠券（优惠券区别与正常的营销活动，是一种独立的促销方式）

前台用户

  1. 用户通过app或h5页面浏览商品信息，其中参与交易的营销和售卖信息通过网关层统一聚合，返回给页面展示
  2. 用户浏览完成选择想要购买的商品，下单支付，交易系统收到请求，会通过营销系统验证营销信息，通过商品系统验证商品信息，无误后完成交易行为

2.1 用户下单前查询商品营销价

2.1.1 用户下单前查询商品营销价序列图概述

  2. misc系统组装查询营销活动请求，调用promotion系统查询营销活动
  3. promotion系统根据传入的营销活动查询营销活动，组装营销规则和行为，最后计算营销价格返回给客户端

  2. order系统发起价格验证，首先调用lesson系统获取 sku原价，之后组装计算折扣请求
  3. order系统调用promotion计算营销价格，对比计算后的营销价格和用户下单出入是否相符

3.斑马营销子系统设计

按模块拆分主要分为三部分

  1. 营销活动：管理活动，活动规则，活动方式
  2. 营销活动范围：管理营销活动对应的商品信息
  3. 赠品管理：管理营销活动的赠品

其中营销活动按照业务抽象三个业务域

  1. 活动域：统筹整个营销活动，聚合规则和行为，编排活动计算流程和验证流程
  2. 规则域：负责规则的匹配，引入轻量级高性能aviator辅助逻辑判断，分离了业务规则和系统代码
  3. 行为域：负责营销优惠的计算，按照设定的方式进行一口价，折扣，减规定价格等计算

3.1 斑马营销子系统序列图

3.1.1 查询计算营销价格类序列图

3.1.1 查询计算营销价格类序列图概述

  2. service主要负责流程的编排， storage层负责数据的组装，domain层划分了营销活动域，营销规则域，营销行为域三个大的领域
  3. 营销活动域负责营销活动的验证以及活动主流程的编排
  4. 营销规则域负责组装营销规则，后台设置的规则和真是的下单/查询场景匹配情况，返回对应的行为
  5. 营销行为域负责营销价格的计算逻辑

3.1.2 下单计算营销价格类序列图

3.1.2 下单计算营销价格类序列图概述

  1. 流程上下单是滞后的，下单过程中的优惠价格都是查询时候计算好的，所以下单过程验价是一个逆向推导的流程
  2. 按照规则分类订单（一个活动匹配一个规则，活动叠加规则叠加）重新计算优惠价，最后和传入优惠对比，查看是否一致
  3. 首先加载活动和规则行为数据，其次按照每个订单项的actionResult提取关键信息来重新计算优惠的actionResult（此处如果是来联报匹配一个活动，会进行两次同样的actionResult计算，虽然重复了，但是对最后结果没影响，最后计算得出的两次actionResult是一样的）
  4. 最后整合订单项上的的actionResult结果到订单纬度，与传入的优惠价匹配

概述

营销是一种广泛的商业活动和策略，其主要目标是将产品或服务引入市场，吸引潜在客户并实现销售，同时建立和维护客户关系。营销的核心任务是满足客户需求，促进产品或服务的销售，以实现盈利并支持企业增长。

在电商团队中，增长的转介绍、分销官、投放等各个具体的业务场景都与营销息息相关，营销中心系统设计的目标也就是如何更好更快地支持各上层业务对于营销的相关需求。

营销中心作为基础电商 “商品-营销-交易” 三大售卖基础能力的一部分，主要是由 优惠券、团购、促销 三个子系统构成，对外提供相应底层配置管理的能力，以支撑增长等售卖场景对于价格与赠品的多样化业务需求。

营销中心提供的能力与上下游业务关系：

优惠券发放

转介绍兑换现金券

用户在转介绍的活动中，可以获得钻石，通过钻石能够兑换 50 元课程优惠券（可叠加），兑换后会给用户发放优惠券，最终可用于购买系统课：

系统课转化优惠券

增长业务会根据用户系统课的购买情况，给指定用户发放一张面值 300 元的系统课优惠券，激励用户续报系统课。

优惠券兑换码

运营会与第三方合作售卖课程，形式是制作一批实体的优惠券兑换卡，然后在三方的电商平台进行售卖，用户购买后会收到实体的优惠券兑换卡，卡上面记录了优惠券兑换码的信息，通过在 APP 中录入兑换码可以兑换得到一张优惠券，从而以低价购买课程。

售卖页价格定制

业务通过配置落地页，关联引入价格赠品策略信息，可以实现该落地页售卖的商品价格与赠品的定制化诉求。

  • APP 端内系统课老用户享受 95 折优惠购买系统课
  • 转介绍多科售卖：购买三科比购买两科的价格更优惠，购买两科的价格比单科购买的价格优惠。
  • 扩科低价搭售：购买任意一科体验课之后，扩科购买可享受 9.9 一口价优惠。
  • 投放买二赠一：购买两科体验课的话，可以 0 元的一口价优惠购买第三科。

售卖页赠品定制

  • 系统课常规礼包：对于系统课，可以配置多选一礼包供用户在售卖页中选择
  • 营销策略加赠：营销策略加赠配置则为用户提供了 M 选 N 的礼包

系统设计

以下分别介绍三个系统的相关设计。

一. 优惠券

优惠券是一种具有一定价值的凭证，企业发放给消费者，消费者在购买指定产品或服务时可以使用优惠券抵扣相应的金额或享受特定优惠。

斑马的优惠券分为两种类型，一种是不可叠加的代金券，一种是可相互叠加的现金券。优惠券的优惠形式分为折扣与满减两种，用户在购买课程或者实物商品的时候可以使用优惠券抵扣一部分的订单金额，以达到优惠购买的目的。

系统能力

功能 说明
优惠券模板管理 提供优惠券模板的创建/编辑/查询，可配置券的各种信息：券类型优惠类型及金额使用条件：启用门槛适用用户适用商品过期时间
发放优惠券 运营后台手动发放各业务按照自身诉求在合适的时机给用户发放指定的优惠券
使用与核销能力 支持在下单的时候关联校验与订单绑定的优惠券，且在用户支付成功之后完成券的自动核销
推荐优惠券 可根据用户的购买请求，为用户推荐最优的优惠券组合
优惠券兑换码 运营可在管理后台生成一批兑换码，用户凭借兑换码可以兑换出其关联的用户优惠券
支持各种业务用券 国内课程国内百科服务版、内容版课程海外百科

系统设计

在设计优惠券的时候，我们采取了优惠券模板+用户优惠券的形式，其中优惠券模板承载券无状态的共性，而用户优惠券则承载了用户个性化的信息。同一个模板可以复用，通过模板可批量高效地生成用户优惠券。

优惠券模板

优惠券模板是定义券的基础公共属性的实体，斑马的优惠券模板定义了优惠券的以下信息：

字段 含义
名称 用户在优惠券列表页可见的优惠券名称
描述 用户在优惠券列表页可见的优惠券的详细说明信息
备注 用户不可见的优惠券备注信息，运营可标注券的一些额外信息，如海外场景使用此字段承载券的中文描述信息，便于运营管理优惠券
券类型 确定优惠券的可叠加方式：（代金券与现金券属于两种类别的券，可以同时使用）代金券：不可叠加现金券：可相互叠加
优惠类型 确定优惠金额的计算方式：打折、立减
优惠金额 按照优惠类型的不同，承载折扣值或者立减的具体金额
门槛金额 订单总金额必须达到配置的价格阈值之后，才能使用优惠券
有效期 不限制使用时间，即永久有效指定的绝对时间点过期，如 2023.12.01 23:59:59 过期按照自然日限定过期时间，如收到券后 7 天之内有效
模板状态 启用/停用，用于管控优惠券模板的生命周期
适用用户 只有符合用户身份限定条件的部分用户，才能够使用优惠券
适用商品 限定优惠券适用的订单商品，只有在购买券上关联配置的商品时，才能够使用优惠券：courseIds：指定的 course 可用，国内及服务版课程商品可配置spuIds：指定的 spu 可用，国内百科、海外百科及内容版商品可配置categoryIds：指定的商品类目可用，国内百科可配置
适用商品的订单条件 订单中的商品需满足 等于/包含 X 个券上配置的商品时可用（默认为购买的内容包含任意1个适用内容时可用，目前仅百科可配置）

用户优惠券

用户优惠券是发券后，针对具体用户生成的券，拥有独立的 ID，记录了用户个性化的信息：

字段 含义
模板ID 记录当前用户优惠券关联的优惠券模板，用以关联模板上面配置的适用商品条件等信息
用户ID 记录用户当前优惠券的用户ID
来源 当前的用户优惠券是怎么获得的
状态 是否已经被使用，如果被使用了，也会额外记录使用了当前优惠券的订单ID
过期时间 当前用户优惠券什么时间过期

用户在 APP 中看到并使用的就是用户优惠券，在选定了商品进入下单确认页之后，优惠券系统也提供了一个推荐优惠券的接口，会默认给用户推荐勾选最优惠的优惠券组合。

目前用户优惠券的总量不算多，未过期的券加上过期时间不超过半年的券一共 2000W 左右，定期归档的话可以维持在一个较低的数据量水位线上，目前看来还没有碰到较大的性能问题，但后续业务对于券的依赖较重的话，可以考虑对用户优惠券表作分库分表处理，以提升系统性能。

二. 团购

团购是指一组人一起购买商品的活动，以获得折扣或更有利的价格。在团购活动中，涉及到开团、参团、成团等流程：

  • 开团：通过自己成为团购活动的发起人来组织购物团队的方式，以便获得折扣。开团之后要求一定的社交互动和努力来吸引足够多的人参与，以达到团购折扣的要求。
  • 参团：参与一个已经形成的购物团队或团体，以获得团购折扣。
  • 成团：参团人数达到成团配置的人数之后，拼团成功，此时会触达后续商品购买成功的履约流程。

三. 促销

促销系统是指通过各种营销手段刺激消费者购买的系统。斑马电商的促销系统主要提供的是价格赠品策略平台的能力，可支持特定条件规则下的价格优惠及赠品策略配置。

促销系统的设计考量，在一定程度上借鉴了营销的人货场理论：

人货场理论（3C理论）是一种广泛应用于零售和市场营销领域的理论，用于分析和解释消费者购买决策和零售商的策略。

  • 人（Customers）：这代表着消费者，也就是购买产品或服务的人群。消费者的需求、偏好、购买行为、购买力等因素都在这一要素中起作用。了解消费者是市场营销策略的核心，包括确定目标市场、了解其需求和制定针对性的推广计划。
  • 货（Commodities）：这指的是商品或服务，也就是零售商销售的产品。这一要素包括产品特性、品质、价格、供应链管理等。企业需要确保其产品或服务能够满足消费者的需求，具有竞争力，并提供有吸引力的价值主张。
  • 场（Channels）：这是指产品或服务的销售渠道，包括销售地点、分销渠道、推广途径等。企业需要选择合适的销售渠道，以确保产品能够有效地传达给目标消费者，满足其需求。
  • 选人：在需要营销的场景中，选人这一步由促销活动的上层来完成的，如落地页来决定需要给哪些用户曝光该落地页，促销系统内部实现的是对于不同人群的定价策略
  • 选品：哪些商品能够参与此次促销活动，斑马商品品类较少，目前都是在活动上面按照商品 ID 配死参与的商品是哪些
  • 选场：决定在什么渠道进行此次促销活动，斑马目前均是由上层业务自行指定各自场景需要开展怎样的促销活动，促销系统也暂未将渠道视作强制约束的条件

系统能力

功能 说明
B 端管理 支持配置促销策略的规则、策略上下线等
实时的促销计算 根据用户购物车中的商品，以及参与的营销策略活动，实时计算优惠价格及其可享受的加赠品
整单优惠 购物车中的商品只要满足了促销配置的规则条件要求，则可基于整笔订单按照立减/打折/一口价的配置情况，对总价进行优惠，最后再将优惠分摊至每一个商品上
指定商品优惠 购物车中的商品只要满足了促销配置的规则条件要求，则可按照配置的指定商品的优惠形式，不作优惠分摊，仅指定其中某些商品的售价优惠信息

常规礼包赠送 在用户购买特定的系统课时，按照用户的身份条件，为其附加符合要求的常规礼包加赠品
课程/商品赠送 提供了为用户直接赠送课程/实物/百科商品的能力

系统设计

促销系统的设计初衷是为了能够更快的支持业务对于多元化营销规则的诉求，所以在设计之初对于系统的可扩展性考量较多，促销活动的整个流程其实是比较清晰的，即用户给予参与活动的 ID、商品信息，然后促销系统内部依据这些信息去匹配活动配置的各种优惠规则条件，如果没有命中任何条件，则表明用户无资格享受任何优惠；如果按照优先级命中了活动的某一个规则条件，则执行该规则关联的优惠动作，即可以享受的优惠价或者加赠品。

促销系统的实现时，考虑到规则的组合多种多样，硬编码的方式实现起来不好维护，所以引入了轻量级的规则引擎：Aviator，借助它实现了可能的复杂多变的规则配置，也正是因为此，后续如果产品的诉求是基于商品或者用户迭代策略规则的限制条件的话，研发侧的修改也是比较简单的，只需要实现一个新的规则变量提取器就行了，相比于硬编码的方式，能够极大地缩短研发工期。

Aviator 是一个高性能、轻量级的 Java 语言实现的表达式求值引擎，主要用于各种表达式的动态求值。Aviator 没有通过解释的方式运行，而是直接将表达式编译成 Java 字节码，交给 JVM 去执行。

规划

长期来看，营销中心系统总体都是围绕的人、货、场领域，基于此设计的一些优惠实现方式，伴随业务的发展尽可能的贴合业务，没有过度设计，但是在系统底层能力上面其实可以抽象一些通用的领域能力：

  • 对于团购活动而言，其实本质上就是一个促销活动，后续已经有规划将团购活动纳入促销系统中统一维护，可借助促销系统内部的规则引擎发挥更多的团购系统能力；
  • 优惠券其实也是依据一定的用户/商品条件，判定用户的券是否可用，以及条件匹配之后，按照配置的优惠金额为用户的订单进行优惠，所以后续也可把优惠券的适用条件抽象化为规则条件，以接入更加灵活可管理的券适用条件的配置能力；
  • 目前优惠券系统不感知任何业务逻辑，券发放全部由上层业务自行控制，如果营销系统后续进行资产预算管理的话，发券的行为可以活动化处理，即将领券/发券行为视作一个促销活动来管控
  • 兑换能力可以统一抽象一下，而不局限于兑换优惠券，可以支持用户直接根据兑换码获得商品/课程
  • 营销中心也可以方便的接入触达能力，比如给用户推送触达提醒优惠券即将过期或者促销活动即将结束
  • 促销系统欠缺数据统计分析的能力：哪些策略在使用，使用的情况是什么样的，如 xx 策略的访问量，及其实际成单量的统计
